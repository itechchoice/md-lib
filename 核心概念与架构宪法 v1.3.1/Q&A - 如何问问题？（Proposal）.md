# **✍️ Q&A 页统一问题措辞风格（Proposal）**

**这里的问题不是为了找“实现位置”，而是为了澄清“概念边界与设计裁决”。**

## **一、问题的总体要求（必须同时满足）**

每一个 Q&A 问题，**至少要隐含或显式包含这三点中的两点**：

1. **语义范围**
    
    👉 在问 *哪一类概念*（Context / State / Authority / Decision 等）
    
2. **生命周期 / 时间边界**
    
    👉 在问 *什么时候存在、什么时候终止*
    
3. **架构裁决目标**
    
    👉 在问 *我们为什么要这样设计 / 避免什么风险*
    

> ❗**不满足以上要求的问题，优先视为“需要改写的问题”，而不是“需要回答的问题”。**
> 

---

## **二、推荐使用的标准问题句式（强烈建议）**

### **✅ 模板 1｜「并置张力型」（最推荐）**

用于 **两个看似矛盾的设计点**。

> 在我们的架构中，A 是必须的，但同时又明确禁止 B。
在架构上，我们是如何划分 A 与 B 的边界 / 生命周期，以避免 C 风险的？
> 

**📌 示例：**

Interaction Surface 需要管理 Context，但 Execution Plane 又明确禁止 Context。在架构上，我们是如何划分 Context 的合法范围与生命周期的？

---

### **✅ 模板 2｜「生命周期型」**

用于澄清 **一个概念什么时候合法、什么时候非法**。

> X 在系统中的合法生命周期是什么？在哪个节点必须被终止或冻结？为什么？
> 

**📌 示例：**

Context 的合法生命周期是什么？为什么它必须在 Workspace 创建时终止？

---

### **✅ 模板 3｜「裁决目标型」（偏架构）**

用于强调 **反 Agent / 防越权 的设计目标**。

> 在一个明确禁止 Agent Runtime 的架构中，我们为什么要对 X 做出这样的限制？这样做主要是在防止哪类系统演化风险？
> 

**📌 示例：**

在一个禁止 Agent Runtime 的架构中，为什么 Execution Plane 必须是“愚钝”的？

---

## **三、明确不推荐的提问方式（需要改写）**

以下问题 **不是“不允许”，而是“不直接收录”**，建议先改写。

### **❌ 类型 1｜“放哪里型”**

> Context 应该放在哪一层？
Retry 应该写在哪个模块？
> 

**问题：**

- 假设“一定有一个归属”
- 忽略生命周期与裁决目标

**👉 改写为：**

**Context 的合法生命周期如何界定？为什么它不能进入 Execution？**

---

### **❌ 类型 2｜“实现直觉型”**

> 那我能不能在 Workspace 里顺便做 retry？
Orchestrator 能不能看一下结果内容再决定？
> 

**问题：**

- 从实现便利性出发
- 容易天然导向 Agent 行为

**👉 改写为：**

**Retry / fallback 属于哪一类权力？为什么它只能发生在 State Orchestration Plane？**

---

### **❌ 类型 3｜“是或否型”**

> Execution 里能不能有 Context？
Workspace 能不能判断结果？
> 

**问题：**

- 容易得到简单 Yes / No
- 不利于沉淀设计动机

**👉 改写为：**

**为什么 Execution Plane 中 Context 被视为非法概念？如果存在会带来什么系统性风险？**

---

## **四、Q&A 页面推荐的「标准问题格式」**

```
Q&A 提问格式建议：

- 问题应聚焦于概念边界、生命周期或设计裁决动机
- 避免仅以“放在哪里 / 能不能 / 可不可以”为问题核心
- 鼓励使用“为什么 + 如何划分边界”的方式提问
```

---

## **五、提问心法**

> 不要问「X 在哪里」
**要问「X 在什么时候开始生效、在什么边界内存在、又在什么时候被强制终止」。**
> 

这是整套架构宪法的**灵魂提问方式**。
